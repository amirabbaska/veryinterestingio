      Is this so?

      :) great question. And no and yes. The bios used to be a only bootstrapper to enable the computer to boot from any of the IO devices. So the bios would start, do the self-test routine (POST), and launch some form of bootloader. The "bios" and hardware variables would then be set once in the hardware interpretation layer, and would never actually have any sort of api intended for program or runtime access towards itself. Any system variables would be set via environmental switches that then some device-driver or other would take into account on the program interpreter layer (so that you'd have, in this abstraction, hardware layer->bios->OS<->program layer, with the program layer containing certain system-functions that would access the hardware layer on specific addresses).

      Later (with for example acpi for switching devices on and off or putting the computer to sleep in various different states, waking it up again), an amount of these functions have become either abstracted away into more (or less) reliable OS functions (see: IRQ). Or else removed altogether, by convention, or through a more sane and secure platform philosophy. In the sense that you can still access direct memory areas, and allow a program to do it without checking if you're making a mess in the process. But certain areas are occupied by reserved OS-layer functions. And on the other end, accessing protected bios functions or memory areas (which should rather be more correctly called "firmware", like you say) is going to be treated by said firmware logic running in the interpreter layer next to the hardware layer.

      That then now all of a sudden functions as an interpreter layer for higher level abstractions and processes, rather than just a bootstrapper.

      So the question here is, I guess, really what happens when you just turn the power on from the psu. And what happens when you decide to start the boot. Which is that when the power on the mainboard turns on, some amounts of basic routines will start, usually on their own integrated circuits. Switches will be electrified so they can be semiconducted, or something along those lines. And some of the logic that used to be ran once the bios booted is actually running at this point (such as listening to certain events on the bus for, say, wake-signals, errors, etc). Depending on which year we're talking now, ram can be maintained but not yet read, or parts of the bus can read parts of the memory areas, etc.

      And this is typically the state the computer will enter when you turn it off via (electronic) switch, or via software command.

      Meanwhile, the bios still will launch it's own self-test and then another layer of for example an EFI-boot, that then in turn can load any amount of OSes, or Windows via it's own bootloader that becomes another abstraction layer again.

      So yes, the bios still exists in a sense, and it will run self-tests and provide bootstrap functionality and IO control. But some if not most of these routines are actually run the instant the 5v current is put on the mainboard nowadays, which happens when you hit the power-switch on the psu.
