      • OneAndOnlyJackSchitt

        In the case of software I've maintained, the user was never meant to see the error code.

        The class library threw exceptions when a function was called with incorrect values, per proper design. The app calling the function assumed it would always call the function with correct values due to the values being sanitized (the input fields would not accept incorrect value).

        Of course, someone else develops code that calls this function but fails to sanitize the fields. It makes it past the idiots in QA and the client starts getting an occasional "An error has occurred: " followed by the app closing.

        The developer didn't bother to implement exception filtering for the specific exception thrown by the class library and this exception didn't have a field for display text, is was simply "ClassLibraryName.ValuesOutOfRangeException". So the default exception handler took over and displayed a message box that said "An error has occurred: " and executed the Finally block of the default exception handler.

        Tl;dr: There's so many people and departments involved in producing code. There's many many opportunities for errors ("exceptions" in programming terms) to be generated and many many opportunities for errors to be caught and displayed to the user in a meaningful way. A god QA team will catch most of the errors and get the developers to include friendly text with the error. Once in awhile, though, there's an error which isn't expected for some reason. These are the ones which either have really esoteric text which is only meaningful in the specific context of where the error occurred, or which only display a number, or which lead to the dreaded "Unhandled exception has occurred in your application."

    The use of error codes is considered a good practice in programming, because it allows different errors to be strictly defined and distinguished from one another.

    But you don’t have to use error codes specifically. In languages capable of exceptions you’d use different exceptions (which are hopefully also much more meaningful than some error code).
