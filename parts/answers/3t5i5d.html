Koooooj A:

Yes, technically, but in practice no. Also, Brute Force wasn't the issue in the first place.

When you're dealing with an online system it is incredibly easy to defeat a brute force attack: just limit the rate at which people can try to log onto an account, or limit the number of tries they get before they have to wait a long time or go through a more extensive process.

The bigger security hole is people using a super simple, obvious password, like "password" or just their username. These are the passwords that are easy enough to guess that an attacker can occasionally get in through the front door.

Additionally, even if the hacker manages to get an encrypted database of passwords that they want to use a Brute Force attack on then they'll have a harder time when there are reasonable restrictions placed on the passwords. For example, if there's no requirements placed on the password then I'd start with a dictionary attack: guess every word in all lowercase letters. A lot of people will survive this attack, but plenty won't. If there was a requirement for a number then I now have to figure out what number the lazy users used (9 times out of 10 it's a 1) and where they put it (almost always in the first or last spot). Same thing when there needs to be a symbol (it's going to be ! most of the time), or there needs to be a capital letter (it's the first letter).

By adding restrictions there are now a bunch of passwords that don't need to be guessed, but the minimum complexity of passwords has gone up so I have to work harder to guess them. Remember: the hacker doesn't have to search through every possible password; they just have to guess a bunch of passwords that the least security conscious people used.

Ultimately password rules tend to be poorly implemented, but they at least have a theoretical background that makes some sense. The better approach to password rules would be to implement a minimum length, since length is the ultimate counter to brute force. Unfortunately many sites go the opposite direction and make a maximum length instead, but that's likely because they incompetently coded part of their site and can't handle larger inputs.

Schnutzel A:

The number of passwords that conform to these limitations is far greater than the ones that don't. When disallowing the "weak" passwords, you are still left with many more "strong" passwords.

For example, an 8 character password made up of only lower case letters has 26^8 = 208827064576 combinations. Allowing numbers gives you 36^8 = 2821109907456 combinations, which about 13.5 times more options. If you disallow passwords without numbers, you are still left with 12.5 times more combinations.

eqleriq A:

Top comment doesn't answer the question, here is the answer:

OP is contradicting themselves a bit.

Brute force goes through every combination. Dictionary uses popular patterns / common terms first.

So, saying "brute force dictionaries" is a bit of a contradiction. The idea is that patterns are more common than not (entropy). If you have four numbers in a row, 1234 would be more common than 3412. 2345, 3456, 4567 etc. When you configure a dictionary you can have it insert or move around the "rules" like

[3 letter word][4 number run][symbol] this would do cat1234! dog1234@ man3456# and so on.. (obviously with more order)

So by setting up a rule "no 3+ character runs like ABC abc 123" or "No common dictionary words" or "do not put the name of the website in the password" you are reducing the dictionary without making a meaningful dent in the brute force numbers. Removing something a dictionary might find in an hour that makes a brute force require 999 years instead of 1000 is a great trade off

you could then append [try most common password dictionary] [try dictionary with first letter capitals] [try dictionary with first letter capitals and symbol/number for common substitutions]

apple

Apple

App1e

App13

App!3

App!#

Apple$

and so on...

that is a "hybrid" attack.

Think of it this way:

there are only lower case letters like this 26^8 = 208827064576

But how many 2+6, 3+3+2, 5+3 etc WORD combinations are there? Way, way less. So you can use a dictionary to go through all of those combinations really fast.

So saying "your password must not include runs of numbers" is because a dictionary attack will rapidly try runs like 1234 attached to common words.

cat1234 is far more common a password than c1at234 which is more common than c1a23t4 etc.

brute force would take a fairly similar amount of time to find each one.

but cat1234 with a properly configured dictionary attack is VERY simple to find.

Also, keep in mind that these stats are easy to determine by parsing the "millions of passwords stolen" databases. You can see how common certain patterns are and configure your dictionary attack to follow those patterns.

TLDR: bruteforce != dictionary, those rules make brute force trivially more efficient but dictionary/hybrid attacks way, way, way less feasible

If you do the maths you actually still get an increase in the number of passwords when forcing to conform to restrictions compared to simple lower case passwords. Here are a few examples (restrictions used are: password contains a number, a symbol and an upper case character):

8 characters: ~5000 as many 'restricted' passwords as lower case ones

10 characters: ~10^5 as many passwords

12 characters: ~10^6 as many passwords

So you see there is a benifit. Especially since many people do not see a need for safe passwords and would probably resort to using short simple passwords if not forced to choose better ones.

Of course the security could also be greatly improved by simply choosing long lower case passwords.
